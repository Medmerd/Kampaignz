# BattleScribe XML to Postgres Data Importer

This package contains the logic to parse `.gst` (Game System) and `.cat` (Catalogue) files generated by BattleScribe and ingest them cleanly into a normalized PostgreSQL database schema.

## Architecture

The parsing relies on traversing the complex XML hierarchy and flattening it into relational JSON structures, which are then batch-inserted into Postgres.

### How XML Maps to Database Tables

| XML Tag / Element | SQLite Table | Description |
| :--- | :--- | :--- |
| `<gameSystem>` | `game_system` | The root game system definitions (e.g. Warhammer 40k). |
| `<catalogue>` | `game_catalogue` | Faction-specific files that inherit the system (e.g. Custodes). |
| `<costType>` | `game_cost_type` | Schema meta-definitions for costs (e.g., pts, PL, CP). |
| `<profileType>` | `game_profile_type` | Schema definitions for profiles (e.g., "Weapon", "Unit"). |
| `<characteristicType>` | `game_characteristic_type` | The specific fields for a profile type (e.g., "Range", "M"). |
| `<categoryEntry>` | `game_category` | High-level keywords/categories (e.g., "Character", "Vehicle"). |
| `<selectionEntry>` | `game_selection` | The actual models, weapons, or wargear items. |
| `<selectionEntryGroup>` | `game_selection_group` | Containers for selections where choices must be made. |
| `<forceEntry>` | `game_force_entry` | Detachments, formations, and army organization charts. |
| `<rule>` / `<sharedRules>` | `game_rule` | Textual rules describing game mechanics. |
| `<profile>` | `game_profile` | An instance of a rule/stat block bound to a selection. |
| `<characteristic>` | `game_characteristic` | A specific stat value (e.g. "6" for M) tied to a profile. |
| `<cost>` | `game_cost` | The specific point value assigned to a selection. |
| `<modifier>` | `game_modifier` | Conditional changes to stats or points (e.g., "-1 AP"). |
| `<modifierGroup>` | `game_modifier_group` | Logical groupings (`AND` / `OR`) for sets of modifiers. |
| `<condition>` | `game_condition` | The "if" statement trigger for a modifier or constraint. |
| `<conditionGroup>` | `game_condition_group` | Logical groupings (`AND` / `OR`) for sets of conditions. |
| `<constraint>` | `game_constraint` | Roster building limits (e.g., "Max 3 of this selection"). |
| `<entryLink>` | `game_entry_link` | A pointer referencing a global `<selectionEntry>`. |
| `<categoryLink>` | `game_category_link` | A pointer referencing a global `<categoryEntry>`. |
| `<infoLink>` | `game_info_link` | A pointer referencing a global rule or profile. |
| `<catalogueLink>` | `game_catalogue_link` | A pointer allowing one catalogue to inherit from another. |
| `<infoGroup>` | `game_info_group` | Groups of rules or profiles to organize complex entities. |

### Optional Attributes
In addition to the core IDs and names, the parser accurately extracts missing optional attributes from the XML such as `hidden`, `collective`, `import`, `type`, `scope`, `shared`, `sortIndex`, and `defaultAmount`, storing them as dedicated columns within these tables to maintain full parity with the `.gst` and `.cat` source files.

### Handling of IDs and Fallbacks

BattleScribe is a highly relational structure relying heavily on unique `id` and `targetId` strings.
1. **Explicit IDs**: Core entities (`game_selection`, `game_profile`, `game_characteristic`, `game_rule`, `game_link`) *must* have an explicitly defined `id` in the XML. If the XML is malformed and omits the ID for these items, the parser will **skip** the item to prevent orphaned/corrupt database references.
2. **Auto-Generated UUIDs**: Strictly nested logic entities that contain values but are never referenced externally by `targetId` (like `<cost>`, `<modifier>`, `<constraint>`, and `<condition>`) often do not have native XML IDs. For these specific tables, the parser automatically generates a random UUID (`crypto.randomUUID()`) to act as the primary key in Postgres.

### Inheritance: Shared vs Local

- Elements defined directly within the `<gameSystem>` have `catalogue_id` set to `null` in the database.
- Elements defined within `<shared...>` blocks inside a `.cat` file have their `parent_id` set to `null` and are available globally to the catalogue.
- Elements explicitly nested under a specific `<selectionEntry>` have their `parent_id` set to the ID of that selection.

## Running the Importer

```bash
VITE_DB_CLIENT=devpostgresql npm run start:xml
```

The ingestor operates deterministically using Transactions:
1. It reads the `<gameSystem>` or `<catalogue>` `id` and `revision`.
2. It executes a `DELETE` command across all mapped tables for that specific `game_system_id` and `revision`.
3. It performs a bulk `INSERT` of the newly parsed structure.
